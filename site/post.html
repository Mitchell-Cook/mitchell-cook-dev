Codifying a whole server in one repo

July 8, 2026 · 4 min read

  • infra
  • devops

This site is its own source of truth. The website and the server it runs on live in one repo: push to main, and a GitHub Action applies the change to a single DigitalOcean droplet. One box, one domain, edit-here-and-it-goes-live.

Two deploy paths, split for safety

The trick is separating the safe, frequent changes from the sensitive, rare ones. Every push runs apply.sh, which only touches the site and nginx — validated, then reloaded: 

#!/usr/bin/env bash
set -euo pipefail

rsync -a --delete site/ /var/www/mitchellcook.dev/
cp infra/nginx/site.conf /etc/nginx/sites-available/default
nginx -t                # validate before doing anything
systemctl reload nginx  # zero-downtime reload

Anything sensitive — sshd, the firewall, fail2ban, TLS — lives in provision.sh, which is only ever run by hand. It never gets wired into automation.

If a script can lock you out of the box, a robot shouldn't be allowed to run it on a whim.

Why one repo

  • The config that runs the server is versioned alongside the site.
  • Rolling back a bad change is just git revert.
  • There's exactly one place to look when something breaks.

It's boring on purpose — and boring is exactly what I want from the thing that keeps the lights on.
